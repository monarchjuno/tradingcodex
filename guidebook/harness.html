Core concept

The harness

The harness is the runtime contract generated into an attached workspace. It makes the skills, fixed roles, service boundaries, workspace records, and audit trail work together without turning the workspace into a black box.

Three connected planes #

TradingCodex uses a clear division of responsibility. Each plane has a different job, and none substitutes for another.

PlaneOwns
Codex control planeHead Manager instructions, fixed-role configuration, skills, and dynamic role orchestration.
Service planeMCP identity and capability checks, artifacts, policy, orders, approvals, broker execution, and audit.
Workspace system planeGenerated configuration, run records, research Markdown, source snapshots, audit files, and local launchers.

Research orchestration is Codex-native. The service does not classify an investment request, select a team, compile a DAG, or schedule analyst tasks.

What attach creates #

tcx attach projects the workspace contract into a folder you own. It includes Head Manager instructions, fixed-role TOML, hooks, the available skill bundles, local policies and run state, research folders, and the tcx launcher.

  • Readable workspace files. Research, source snapshots, Decision Memory, non-private provenance, Strategies, and Brain sources can remain version-controlled.
  • Private state stays local. Runtime databases, credentials, private Investor Context, and transient process state are excluded by the managed privacy block.
  • No automatic publication. Attach and update do not stage files, create a commit, configure a remote, or push.

Useful tools, separated authority #

Normal analysis uses the trading-research permission profile. Roles may use shell, Python, command-line data tools, credential-free public sources, and user-owned files outside trading/. Disposable intermediates use the dedicated $TRADINGCODEX_SCRATCH path rather than the whole temp tree. More-specific rules keep trading/, generated control files, Git metadata, launchers, credentials, and TradingCodex runtime/database state read-only or denied; the native network proxy permits public destinations while blocking local/private services.

Controlled trading/, connector, and managed lifecycle changes require a fresh root task with trading-build selected and $tcx-build as the exact first line. Ordinary reviewable file edits outside trading/ do not. Build permits workspace-local code and tests but disables network access and preserves the protected-state denials. Orders, approvals, broker access, and durable research records stay behind typed service calls in either profile.

What the hooks enforce #

Hooks bind a native task to its workspace and run, validate exact fixed-role child shape, and enforce the analysis-only tool posture. They also recognize the literal, bounded syntax for Build and final order paths and record redacted hook and subagent events.

Hooks do not decide investment meaning. Outside exact reserved syntax, they do not infer a universe, select roles, interpret negations, or turn prose into a semantic plan. Head Manager reads the user mandate and preserves those constraints directly.

This distinction lets ordinary natural-language requests work in any language without relying on a brittle keyword router, while the sensitive paths stay exact and deterministic.

Why artifacts matter #

Every producing role writes an authenticated research artifact. The service verifies the role identity and run binding, derives hashes, and records which earlier artifacts were consumed. Head Manager can synthesize only from verified run-local inputs.

The result is a traceable research trail: source/as-of posture, uncertainty, handoff state, and the evidence behind the final synthesis remain visible after the task ends. The browser viewer exposes that information read-only; it does not launch Codex, create runs, or mutate skills.
